# Knowledge-Base Authoring Log

How each `knowledge_base/` document was produced — methodology (why/from what) and
implementation (which code/commands). Companion to plan §2.9. Purpose: a reviewer
(or future you) can trace every KB fact to its source and reproduce the generated ones.

## The governing method (plan §2.9, applied)
Each fact was routed by **provenance**:
- **Machine-checkable source → generated by script** (never hand-typed).
- **No source of truth → drafted from a cited reference, then gated** by
  `scripts/check_kb.py` (consistency) and, for physics numbers, the simulator.

Every fictional proper noun lives once in `knowledge_base/entities.yaml`; every
document is validated against it.

## Tooling built (all under `scripts/`, run in the `xai` conda env)
| Script | Role | Reproduce |
|---|---|---|
| `export_topology.py` | Generates `topology/ieee14_grid.md` from pandapower `case14` | `python scripts/export_topology.py` |
| `kb_extract.py` | Pulls text (optionally keyword windows) from reference PDFs so prose is paraphrased from actual source text, not memory | `python scripts/kb_extract.py <pdf> --grep "<phrase>"` |
| `check_kb.py` | Five-rule consistency gate (§2.9) | `python scripts/check_kb.py --strict` |
| `simulate_consequences.py` | Machine-derived physical consequences (open breakers → power flow → violations/islanding) | `python scripts/simulate_consequences.py --all` |

## Bugs the process caught (self-correcting checks working)
- `check_kb.py` rule 4 was over-strict (required unique MITRE IDs) — but DoS is
  legitimately shared by several attacks; relaxed to format-validation only.
- rule 2 misread backticked **file paths** as feature names; narrowed to
  Title-Case tokens.
- `export_topology.py` hardcoded a 1.06 p.u. voltage ceiling the solved base case
  (1.090) violated; corrected the stated limit to 0.94–1.10 and regenerated.
