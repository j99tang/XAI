# KB Author

Rulebook for turning references into `knowledge_base/` documents that LightRAG
ingests into one coherent graph. Load this before authoring any KB file so the
rules survive across files and sessions. Full rationale: `docs/IMPLEMENTATION_PLAN.md`
§2.8–2.9; provenance record: `docs/kb_authoring_log.md`.

## The one decision: provenance of each fact
Before writing anything, classify every fact:
- **Machine-checkable source of truth exists → GENERATE, never hand-type.**
  Topology (pandapower `case14`), model card (`models/metrics.json`), physical
  consequences (`scripts/simulate_consequences.py`), IP coverage (the CSVs).
  Editing these by hand can only introduce errors.
- **No source of truth → DRAFT from a cited reference, then GATE.** Playbooks,
  attack narratives, feature interpretations, vendor prose. Paraphrase from the
  actual reference text (extract it with `scripts/kb_extract.py` — never from
  memory), then pass `scripts/check_kb.py --strict`.

## Non-negotiable rules
1. **One fact, one file.** State a fact in exactly one document; everywhere else
   cross-reference it. Duplicates drift and make hallucination un-scorable.
2. **Names come from `knowledge_base/entities.yaml`.** Every device, breaker,
   zone, vendor. Adding a new entity means editing the registry first. Never
   invent a name inline.
3. **Evidence boundary (the §2.8 trap).** The model sees only flow statistics —
   no ASDU/IOA/protocol content. A doc may state protocol facts as *background*
   ("IEC-104 uses port 2404") but must never let a narrative attribute an
   *observation* to evidence the model cannot see ("malformed ASDU sent"). Write
   fingerprints in feature-dictionary terms.
4. **Physics claims are simulator-gated.** Any quantitative grid claim must be
   reproducible by `simulate_consequences.py`. Tag base-case numbers as such.
5. **Cite, don't copy.** Paraphrase public sources; never paste copyrighted text
   into a KB file. Reference PDFs live in `references/source_library/` (git-ignored).
6. **Dense, not long.** < ~2 pages each. Every extra doc is retrieval noise that
   can hurt scores.
7. **Tiering.** Build ★ core docs only; ○ deferred docs (protocol_reference,
   protection_control, vendor_manuals, model_documentation) wait until a Phase-4
   CPAS gap justifies them.

## Workflow for a new/updated doc
1. Identify the KB folder and CPAS/persona it serves (plan §2.8 catalogue).
2. If it introduces entities, add them to `entities.yaml` first.
3. Extract source text: `python scripts/kb_extract.py <pdf> --grep "<phrase>"`.
4. Draft (generated bucket → run the script; drafted bucket → paraphrase + cite).
5. Add a `**Cross-references:**` line linking related docs by path.
6. Run `conda run -n xai python scripts/check_kb.py --strict` — must pass.
7. Record method + source in `docs/kb_authoring_log.md`.
8. If entity names changed, re-ingest the LightRAG graph and re-check §3.1b.

## Templates
- **Generated doc:** header banner "GENERATED by `<script>` — do not hand-edit",
  the tables, a pinned tool version. See `topology/ieee14_grid.md`.
- **Drafted doc:** 1-line purpose, a grounding note naming the source(s) and the
  paraphrase/cite rule, the content, a cross-references line. See
  `attack_taxonomy/attacks.md` (primary-source pattern) or
  `regulatory_compliance/compliance.md` (standards pattern).
